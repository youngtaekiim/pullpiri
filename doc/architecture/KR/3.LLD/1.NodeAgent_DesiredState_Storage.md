# NodeAgent Self-Healing - Desired State 저장 및 컨테이너 실행

## 0. 문서의 목적
- **목표**: ActionController로부터 컨테이너 실행 요청을 받아 desired state를 메모리에 저장하고 컨테이너를 실행하는 기능
- **왜 필요한가**: Self-healing의 기준이 되는 desired state를 메모리에 유지하고 초기 컨테이너를 실행해야 함
- **중요**: apiserver가 이미 ETCD에 package/model 정보를 저장하므로, NodeAgent는 메모리 캐시만 사용
- 이 문서는 NodeAgent의 grpc/receiver/actioncontroller.rs에 desired state 메모리 저장 기능을 추가하기 위해 작성되었습니다.
- 이 기능은 이 문서에 포함된 조건 및 규칙들을 따라야 합니다.

## 1. NodeAgent의 기능
- 컨테이너 실행 및 관리: Podman API를 통해 컨테이너 생명주기 관리
- 상태 모니터링: 컨테이너 및 노드 리소스 정보 수집
- **Self-Healing**: Desired state와 Actual state 비교하여 자동 복구 (신규)

## 2. ActionController Receiver의 구현 구조
- **grpc/receiver/actioncontroller.rs**: ActionController로부터 workload 요청 수신 및 처리
- **manager.rs**: 전역 메모리 캐시 `Arc<Mutex<HashMap<String, DesiredState>>>` 관리
- **desired_state.rs**: DesiredState 구조체 정의
- **runtime/podman/container.rs**: 컨테이너 start/stop/restart 함수 (기존)

## 3. Desired State 저장을 위해 ActionController Receiver에 구현되어야 하는 것

```
+-------------------+         +---------------------+         +-------------------+
| ActionController  |  gRPC   |   NodeAgent         | Podman  |      Podman       |
|   (START 요청)    | ------> | (메모리 캐시 저장   | ------> | (컨테이너 실행)   |
|                   |         |  + 컨테이너 실행)   |         |                   |
+-------------------+         +---------------------+         +-------------------+
                                        |
                                        v
                              HashMap<String, DesiredState>
                                    (메모리 캐시)
```

- **인터페이스**: 외부 인터페이스(gRPC)로부터 수신, Podman API 호출
  - **입력**: ActionController로부터 `WorkloadRequest { action: START, pod_info, restart_policy, probe_config }` 수신
  - **처리**: 
    1. `DesiredState` 구조체 생성 (pod_info, restart_policy, probe_config 포함)
    2. 메모리 캐시에 desired state 저장 (`desired_states_cache.insert(pod_name, desired_state)`)
    3. Podman API로 컨테이너 실행
  - **출력**: 메모리 캐시에 저장 완료, 컨테이너 실행 결과 반환

## 4. 지켜야 할 규칙
- **메모리 캐시 구조**: `Arc<Mutex<HashMap<String, DesiredState>>>`
  ```rust
  // manager.rs에서 초기화
  let desired_states_cache: Arc<Mutex<HashMap<String, DesiredState>>> = 
      Arc::new(Mutex::new(HashMap::new()));
  
  // ActionController receiver에 Arc clone 전달
  let cache_clone = desired_states_cache.clone();
  ```
- **저장 방식**: 
  ```rust
  // START 요청 시
  let mut cache = desired_states_cache.lock().await;
  cache.insert(pod_name.clone(), desired_state.clone());
  ```
- **gRPC 메시지 형식**: `WorkloadRequest` 구조체 사용
- **에러 처리**: 컨테이너 실행 실패 시 메모리 캐시에서도 삭제하고 에러 반환
- **DELETE 요청**: 메모리 캐시에서 삭제 및 컨테이너 stop/remove
  ```rust
  // DELETE 요청 시
  let mut cache = desired_states_cache.lock().await;
  cache.remove(&pod_name);
  podman::stop(&container_id).await?;
  podman::remove(&container_id).await?;
  ```
- **노드 재시작 시 복구**: manager.rs 초기화 시 apiserver ETCD에서 로드
  ```rust
  // manager.rs main() 초기화
  async fn load_desired_states_from_etcd() -> HashMap<String, DesiredState> {
      let mut states = HashMap::new();
      
      // apiserver가 저장한 model 정보 조회

      match common::rocksdb::get_all_with_prefix(prefix).await {
          Ok(kvs) => {
              for kv in kvs {

                  // DesiredState 생성 및 저장
              }
          }
          Err(e) => eprintln!("Failed to load from ETCD: {:?}", e),
      }
      
      states
  }
  ```

## 5. DesiredState 구조체 정의 및 이유

### 5.1 왜 이 필드들이 필요한가?

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct DesiredState {
    pub pod_name: String,
    pub container_id: String,
    pub restart_policy: RestartPolicy,
    pub probe_config: Option<ProbeConfig>,
    pub created_at: SystemTime,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub enum RestartPolicy {
    Always,
    OnFailure,
    Never,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct ProbeConfig {
    pub liveness: Option<LivenessProbe>,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct LivenessProbe {
    pub probe_type: ProbeType,
    pub initial_delay_seconds: u32,
    pub period_seconds: u32,
    pub timeout_seconds: u32,
    pub failure_threshold: u8,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub enum ProbeType {
    Http { path: String, port: u16 },
    Tcp { port: u16 },
    Exec { command: Vec<String> },
}
```

### 5.2 각 필드의 역할

| 필드 | 타입 | 크기 | 이유 |
|------|------|------|------|
| `pod_name` | String | ~50B | reconciliation loop에서 컨테이너 식별용. HashMap의 key로도 사용 |
| `container_id` | String | 64B | Podman API 호출 시 필요 (inspect, restart, stop 등) |
| `restart_policy` | RestartPolicy | 1B | handle_exited_container에서 재시작 여부 판단 |
| `probe_config` | Option<ProbeConfig> | ~200B | probe_loop에서 liveness probe 설정 참조 |
| `created_at` | SystemTime | 16B | 디버깅 및 로깅용 (컨테이너 최초 생성 시간) |

**총 크기**: 약 **331 bytes per container**

**메모리 사용량 예상**:
- 10개 컨테이너: 3.3 KB
- 100개 컨테이너: 33 KB  
- 1000개 컨테이너: 331 KB

→ **매우 작아서 메모리 캐시로 충분!**

### 5.3 왜 ETCD에 중복 저장하지 않는가?

1. **apiserver가 이미 저장**: `/package/{name}`, `/model/{name}`에 전체 정보 존재
2. **성능**: 메모리 접근이 ETCD보다 1000배 이상 빠름
3. **중복 방지**: 같은 데이터를 두 곳에 저장하면 동기화 문제 발생 가능
4. **노드 재시작**: apiserver ETCD에서 복구하면 됨

### 5.4 왜 Clone을 derive 했는가?

- `Arc<Mutex<HashMap>>` 사용 시 lock을 오래 잡지 않기 위해
- 빠르게 clone 후 unlock하여 동시성 향상

```rust
let desired_state = {
    let cache = desired_states_cache.lock().await;
    cache.get(&pod_name).cloned() // Clone 후 즉시 unlock
};
// 이제 lock 없이 사용 가능
```
