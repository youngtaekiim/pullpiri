## 1. 개요

소스 코드의 무결성과 단일 모듈 개념을 유지하기 위해 브랜치 관리 전략을 정의한다.
브랜치는 크게 **주요 브랜치**와 **보조 브랜치**로 구분되며, 각 브랜치의 생성 기준, 절차 및 담당자를 명확히 규정한다.

---

## 2. 브랜치 유형

### 2.1 주요 브랜치 (Long-lived Branch)

| 브랜치 | 설명 |
|--------|------|
| `main` | 릴리즈 가능한 최신 안정 코드. 공식 베이스라인과 대응되며, 직접 커밋 금지 |

### 2.2 모델 브랜치 (Model Branch)

- **정의**: 고객사에 공식 베이스라인을 제출한 이후, 차량 모델별 맞춤 개발을 위해 생성하는 브랜치
- **생성 시점**: 공식 베이스라인(`main`) 태그 설정 완료 후
- **목적**: 모델별 특화 기능 추가, 고객 요구사항 반영, 모델 간 코드 독립성 확보
- **명명 규칙**: `model/<모델명>` (예: `model/vehicle-x1`)
- **통합 정책**: 모델 브랜치의 공통 개선사항은 별도 PR을 통해 `main`으로 역통합(back-merge) 가능

### 2.3 기능 브랜치 (Function Branch)

- **정의**: 주요 기능 변경 또는 신규 기능 개발 시 생성하는 브랜치
- **생성 시점**: 해당 기능의 이슈가 등록되고 개발이 시작될 때
- **목적**: 기능 단위 병렬 개발, 코드 리뷰 및 테스트의 독립적 수행
- **명명 규칙**: `<유형>/<이슈번호>-<간략한-설명>`
- **통합 정책**: 개발 완료 후 PR을 통해 `main` 브랜치로 머지

#### 기능 브랜치 유형 상세

| 유형 | 설명 | 예시 |
|------|------|------|
| `feat` | 신규 기능 개발 | `feat/123-vehicle-status-api` |
| `fix` | 버그 수정 | `fix/145-state-sync-error` |
| `refactor` | 코드 리팩토링 | `refactor/167-nodeagent-cleanup` |
| `docs` | 문서 작업 | `docs/189-api-documentation` |
| `test` | 테스트 코드 작업 | `test/201-integration-test` |
| `chore` | 빌드·설정 등 유지보수 | `chore/210-ci-pipeline-update` |

## 3. 브랜치 명명 규칙

### 3.1 기능 브랜치
```
<유형>/<이슈번호>-<간략한-설명>
```
- 소문자와 하이픈(`-`) 사용
- 이슈 번호를 반드시 포함
- 설명은 3단어 이내, 영문 소문자 권장

**예시**
- `feat/123-filter-gateway-impl`
- `fix/145-etcd-connection-timeout`
- `docs/167-contribution-guide`

### 3.2 모델 브랜치
```
model/<모델명>
```
- 모델명은 고객사 또는 프로젝트와 협의된 식별자 사용

**예시**
- `model/vehicle-x1`
- `model/ivi-platform-v2`

## 4. 브랜치 생성 기준 및 절차

### 4.1 기능 브랜치 생성 절차

1. GitHub에서 관련 이슈 등록 (이슈 번호 확인)
2. 이슈 페이지의 "Development" > "Create a branch" 이용하거나 명령행에서 생성:
   ```bash
   git checkout -b feat/<이슈번호>-<설명> main
   ```
3. 이슈 상태 라벨을 `status:in-progress`로 변경
4. 개발 완료 후 PR 생성 및 리뷰 요청

### 4.2 모델 브랜치 생성 절차

1. CM 담당자가 공식 베이스라인 태그 확인
2. 모델 브랜치 생성 요청 이슈 등록 (유형: `[TASK]`, 라벨: `type:task`)
3. CM 담당자가 해당 태그 시점에서 브랜치 생성:
   ```bash
   git checkout -b model/<모델명> <베이스라인-태그>
   git push origin model/<모델명>
   ```
4. 이해관계자에게 브랜치 생성 공지

## 5. 브랜치 통합 규칙

### 5.1 기능 브랜치 → main 통합

| 항목 | 기준 |
|------|------|
| 통합 방식 | Pull Request (PR) |
| 리뷰어 | 최소 1명 이상 승인 필수 |
| CI 통과 | 필수 (빌드·테스트·린트 모두 통과) |
| 커밋 전략 | Squash merge 또는 Merge commit (팀 협의에 따름) |
| 브랜치 삭제 | 머지 완료 후 기능 브랜치 삭제 |

### 5.2 모델 브랜치 역통합 (Back-merge)

- 모델 브랜치에서 발생한 공통 버그 수정 또는 기능 개선은 `main`으로 역통합 가능
- 역통합 시 PR을 별도 생성하고, CM 담당자 및 관련 이해관계자 리뷰 필수
- 역통합 전 변경사항이 모델 특화 코드와 분리되었는지 확인

### 5.3 금지 사항

- `main` 브랜치에 직접 `push` 금지
- 리뷰 미완료 상태에서 머지 금지
- CI 실패 상태에서 머지 금지

## 6. 브랜치 담당자

| 브랜치 유형 | 생성 담당 | 통합(머지) 승인 |
|-------------|-----------|----------------|
| `main` | CM 담당자 | CM 담당자 |
| `model/*` | CM 담당자 | CM 담당자 + 이해관계자 |
| `feat/*`, `fix/*` 등 | 개발자 | 리뷰어 (1인 이상) |

---

## 7. 브랜치 보호 규칙

`main` 브랜치에는 다음 GitHub 브랜치 보호 규칙을 적용한다.

1. **GitHub 설정 경로**: Repository > Settings > Branches > Branch protection rules
2. **적용 규칙**:
   - `Require a pull request before merging` (직접 push 금지)
   - `Require approvals`: 최소 1명 이상 승인 필수
   - `Require status checks to pass before merging` (CI 통과 필수)
   - `Require linear history` (히스토리 선형 유지)
   - `Do not allow bypassing the above settings` (관리자 포함 예외 없음)

## 8. 브랜치 관리 다이어그램

```
main ──────────────────────────────────────────────────────►
  │                    ▲              ▲
  │ (베이스라인 태그)   │ PR 머지       │ PR 머지
  │                    │              │
  ├──► model/vehicle-x1 (모델 브랜치)  │
  │       └── 모델 특화 개발           │
  │                                   │
  ├──► feat/123-filter-gateway ────────┘
  │       └── 신규 기능 개발
  │
  └──► fix/145-etcd-timeout ──────────► (PR 머지 후 삭제)
          └── 버그 수정
```

### 브랜치 생애주기

```
이슈 등록
    ↓
브랜치 생성 (개발자/CM)
    ↓
개발 및 커밋
    ↓
PR 생성 및 리뷰 요청
    ↓
CI 검증 (빌드·테스트·린트)
    ↓
리뷰어 승인
    ↓
main 브랜치로 머지
    ↓
기능 브랜치 삭제
    ↓
이슈 자동 종료
```
